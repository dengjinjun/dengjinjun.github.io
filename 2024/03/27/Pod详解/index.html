Pod是Kubernetes中能够创建和部署的最小单元，是Kubernetes集群中的一个应用实例。

Pod可以包含了一个或多个容器，还包括了存储、网络等各个容器共享的资源。Pod支持多种容器环境，Docker则是最流行的容器环境。

pod基本操作

pod相关命令

创建pod

kubectl create -f pod模板文件		# 通过yaml文件创建pod，不支持重复创建
kubectl apply -f pod模板文件		# 同样通过yaml文件创建pod，支持重复执行，适合用于更新资源

查看pod

kubectl get pod		# 查看pod的简短信息
kubectl get pod -o wide		# 查看IP相关信息
kubectl get pod -o pod模板文件		# 查看pod的yaml文件
kubectl get pod --show-labels		# 查看pod的标签

删除pod

kubectl delete pod pod名称		# 基于pod名称的方式删除pod
kubectl delete -f pod模板文件		# 通过yaml文件删除pod
kubectl delete -f .		# 删除当前目录下所有yaml文件创建的pod

创建pod流程

创建单容器Pod

单容器Pod是最常见的部署方式

# 首先编辑一个pod模板
[root@master kubernetes]$ vim /pod/nginx.yaml

# 资源的类型
kind: Pod
# API的版本号
apiVersion: v1
# 元数据信息
metadata:
  # 资源的名称
  name: nginx
  # 自定义资源的标签,其中的KV由你自定义即可.
  labels:
    disk: ssd
# Pod的定义,主要描述这个Pod运行什么服务
spec:
  # 指定容器相关的配置
  containers:
    - name: nginx01  # 容器的名字
      image: registry:80/nginx:v1  # 指定容器的镜像地址
      ports:  # 指定容器的暴露端口
        - containerPort: 80  # 暴露容器的80端口
        
# 然后创建这个pod即可
[root@master kubernetes]$ kubectl create -f /pod/nginx.yaml

# 查看容器状态
[root@master kubernetes]$ kubectl get pod

# 查看容器详细信息
[root@master kubernetes]$ kubectl describe pod nginx

# 连接
[root@master kubernetes]$ kubectl exec -n 命名空间名称 -it pod名称 -c 容器名称 -- command

创建多容器pod

多容器Pod是相对高阶的使用方式，一个Pod内的容器共享IP地址和端口范围，容器之间可以通过 localhost 互相访问。除非应用耦合特别严重，一般不推荐使用这种方式。

# 此处仅给出pod模板，其他操作与创建单容器pod相同
[root@master kubernetes]$ vim /pod/nginx.yaml

# 资源的类型
kind: Pod
# API的版本号
apiVersion: v1
# 元数据信息
metadata:
  # 资源的名称
  name: nginx
  # 自定义资源的标签,其中的KV由你自定义即可.
  labels:
    disk: ssd
# Pod的定义,主要描述这个Pod运行什么服务
spec:
  # 指定容器相关的配置
  containers:
    - name: alpine01
      image: registry:80/alpine:v1
      command: ["sleep","9999"]
    - name: nginx01  # 容器的名字
      image: registry:80/nginx:v1  # 指定容器的镜像地址
      ports:  # 指定容器的暴露端口
        - containerPort: 80  # 暴露容器的80端口

pod模板行为

指定namespace

命名空间是实现环境隔离的

# 查看命名空间，ns是namespace的简写
kubectl get ns		# 通过kubectl命令查看
etcdctl ls /registry/namespaces		# 在etcd数据库中查看
kubectl describe ns

# 创建一个命名空间
kubectl create ns 命名空间名称		# 直接创建
kubectl create -f yaml文件		# 还有一种创建命名空间的方式，是通过文件去创建，类似于通过文件创建pod的方式，可以自行了解

# 删除一个命名空间
kubectl delete ns 命名空间名称

# 使用命名空间
namespace: <string>		# 在pod模板文件中，metadata下级指定`namespace: 命名空间名称`

指定node

指定将该pod调度在某个节点运行

nodeName: <string>			# 在pod模板文件中，metadata下级指定`nodeName:节点名称`

资源限制

reousrces: <object>		# 在pod模板文件中，containers下级指定`resources:资源对象`
resources下级可以指定limits对象和requests对象，前者表示最大资源限制，后者表示最少资源需求
resources:
  limits:
    cpu: 500m		# cpu资源换算是1core=1000m
    memory: 500M
  requests:
    cpu: 100m
    memory: 100M

指定暴露的端口

ports: <[]Object>		# 在pod模板文件中，containers下级指定`ports:资源对象列表`
ports下级可以指定五个对象，示例：
ports:
  - containerPort: 80		# 容器暴露的端口
    protocol: TCP		# 协议，默认是TCP，可选：TCP或者UDP
    name: nginx-01		# 端口名，可以给这个开放的端口命名，后续可以引用
    hostPort: 8888		# 容器的端口绑定到主机的端口，如果指定了使用主机网络，那么要跟容器端口相同
    hostIP: 0.0.0.0		# 绑定到主机的哪个IP地址

使用主机网络

# 使用主机网络类似于docker中的host网络类型，pod跟宿主机共享IP和端口
hostNetwork: <boolean>		# 在pod模板文件中，spec下级指定`hostNetwork: true`即可让pod使用主机网络，默认是false

指定主机名

hostname: <string>		# 在pod模板文件中，spec下级指定`hostname: 自定义主机名`即可指定主机名,如果不指定则是pod名称

重启策略

# pod内所有容器的重启策略，可选Always, OnFailure, Never
restartPolicy: <string>		# 在pod模板文件中，spec下级指定`restartPolicy: 重启策略`，如果不指定默认是always

镜像拉取策略

# 构建容器时的镜像拉取策略，可选Always, Never, IfNotPresent
imagePullPolicy: <string>		# 在pod模板文件中，containers下级指定`imagePullPolicy: 拉取策略`，默认情况下，如果镜像的tag是latest，拉取策略是Always，其他的情况默认都是IfNotPresent

可能遇到的问题

问题1：创建pod时报错：Error from server (ServerTimeout): error when creating “busybox.yaml”: No API token found for service account “default”, retry after the token is automatically created and added to the service account

这是由于apiserver的service account没有设置API token引起的，解决办法：

# 生成密钥
[root@master kubernetes]$ openssl genrsa -out /etc/kubernetes/serviceaccount.key 2048

# 编辑apiserver配置文件
[root@master kubernetes]$ vim /etc/kubenetes/apiserver
KUBE_API_ARGS="--service_account_key_file=/etc/kubernetes/serviceaccount.key"

# 编辑controller-manager配置文件
[root@master kubernetes]$ vim /etc/kubernetes/controller-manager
KUBE_CONTROLLER_MANAGER_ARGS="--service_account_private_key_file=/etc/kubernetes/serviceaccount.key"

# 重启所有服务
[root@master kubernetes]$ systemctl restart etcd kube-apiserver kube-controller-manager kube-schedule

问题2：容器状态一直是ContainerCreating，查看容器详细信息发现有错误：Error syncing pod, skipping: failed to “StartContainer” for “POD” with ImagePullBackOff: “Back-off pulling image "registry.access.redhat.com/rhel7/pod-infrastructure:latest"“

这是由于创建pod所需的基础镜像拉取失败，需要修改基础镜像的拉取地址

# 编辑所有工作节点上的kubelet配置文件，修改拉取基础镜像的地址，默认的仓库国内访问会有问题，所以需要修改为私有仓库
# `registry:80`是我的私有镜像仓库的地址，`pod-infrastructure:latest`是镜像名，如果私有仓库内还没有这个镜像还需要添加镜像
[root@node1 kubernetes]$ vim kubelet 
KUBELET_POD_INFRA_CONTAINER="--pod-infra-container-image=registry:80/pod-infrastructure:latest"
[root@node1 kubernetes]$ systemctl restart kubelet


# 添加镜像，如果私有仓库内已经存在了pod的基础镜像，则不需要此步骤
|root@aliyun ~|$ docker search pod-infrastructure		# 搜索一下有哪些pod基础镜像
|root@aliyun ~|$ docker pull tianyebj/pod-infrastructure		# 选择一个拉取到本地
|root@aliyun ~|$ docker tag tianyebj/pod-infrastructure:latest registry:80/pod-infrastructure		# 打个标签
|root@aliyun ~|$ docker push registry:80/pod-infrastructure		# 推送到私有仓库

pod生命周期

Pending：创建了pod资源并存入etcd中，但尚未完成调度。

ContainerCreating：Pod 的调度完成，被分配到指定 Node 上。处于容器创建的过程中。通常是在拉取镜像的过程中。

Running：Pod 包含的所有容器都已经成功创建，并且成功运行起来。

Succeeded：Pod中的所有容器都已经成功终止并且不会被重启

Failed：所有容器都已经终止，但至少有一个容器终止失败，也就是说容器返回了非0值的退出状态或已经被系统终止。

Unknown：因为某些原因无法取得 Pod 的状态。这种情况通常是因为与 Pod 所在主机通信失败。
